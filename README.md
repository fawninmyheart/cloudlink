Cloudlink is a minimal task bridge for this workflow:

1. The cloud server stores tasks and exposes APIs.
2. A local computer worker polls the cloud server.
3. The worker executes registered task types, including trusted `script_job` payloads in a Cloudlink-only Python environment.
4. The worker reports `success` or `failed` results back to the server.
5. Callers query the server for task status and results.

The server never connects to the local computer, so there is no home-network NAT, public IP, or port-forwarding requirement.

## What Runs Where

Cloud server:

- Receives internal task submissions at `POST /api/internal/tasks`.
- Stores task status in SQLite.
- Lets server-side code query `GET /api/internal/tasks/{task_id}`.
- Serves a protected operations dashboard at `/`.
- Protects worker endpoints with registered worker identity and per-worker credentials.

Local worker:

- Calls `POST /api/worker/claim`.
- Executes explicit handlers for `echo_test`, `generate_daily_report`, and `script_job`.
- Calls success or failed report endpoints.
- Runs dynamic scripts without invoking a shell and keeps their dependencies inside a dedicated Cloudlink virtualenv.

## Security Boundary

Cloudlink `script_job` is trusted remote Python execution, not a sandbox. Run
workers under a dedicated OS user or container, keep secrets out of worker
accounts, and submit only code generated by a trusted cloud-side Codex workflow.
The worker avoids shell execution for payloads, but Python code can still read
files and open network connections allowed by the worker machine.

Cloud-side Codex tokens are intentionally narrower than the internal admin
secret. They can create/query their own tasks and register reusable datasets
only from `CLOUDLINK_ALLOWED_DATASET_SOURCE_ROOTS`. Use the internal admin
secret only in trusted server-side code.

## Codex CLI Deployment Runbook

This project is meant to be installed by Codex CLI on the cloud server after it
reads the repository. Do not maintain a separate deployment skill for Cloudlink.
The repository README is the source of truth.

Cloud-side Codex CLI should explain these requirements before installing:

- A Linux cloud server with `sudo`, `git`, Python 3, `python3-venv`, and outbound
  network access to GitHub.
- A public HTTPS URL is recommended for local workers. A Cloudflare-managed
  domain is fine, but `/api/worker/*` and `/install/worker/*` must not be blocked
  by bot challenges.
- If no domain is available, use an IP-based HTTPS endpoint when possible. Plain
  HTTP worker install is blocked by default because the registration exchange
  returns a worker credential. Enable `CLOUDLINK_ALLOW_INSECURE_WORKER_INSTALL=1`
  only for explicit private testing.
- Nginx or another reverse proxy should expose only the dashboard, worker APIs,
  and worker install routes. Keep `/api/internal/*` local-only.

Install from GitHub on a fresh server:

```bash
sudo apt-get update
sudo apt-get install -y git python3 python3-venv nginx

export CLOUDLINK_REPO_URL="https://github.com/<owner>/cloudlink.git"
export CLOUDLINK_REF="main"

sudo mkdir -p /opt/cloudlink
sudo chown "$USER:$USER" /opt/cloudlink
git clone --branch "$CLOUDLINK_REF" --depth 1 "$CLOUDLINK_REPO_URL" /opt/cloudlink

cd /opt/cloudlink
python3 -m venv .venv
.venv/bin/python -m pip install --upgrade pip
.venv/bin/python -m pip install -r requirements.txt
```

Create `/etc/cloudlink.env`:

```bash
cloudlink_secret() {
  python3 -c 'import secrets; print(secrets.token_urlsafe(32))'
}

WORKER_SECRET="$(cloudlink_secret)"
INTERNAL_API_SECRET="$(cloudlink_secret)"
CLOUDLINK_CODEX_TOKEN="$(cloudlink_secret)"
ADMIN_PASSWORD="$(cloudlink_secret)"

sudo tee /etc/cloudlink.env >/dev/null <<EOF
CLOUDLINK_DATABASE_PATH=/opt/cloudlink/data/cloudlink_tasks.db
CLOUDLINK_DATA_ROOT=/opt/cloudlink/data
CLOUDLINK_PUBLIC_BASE_URL=https://tasks.example.com
CLOUDLINK_ALLOWED_DATASET_SOURCE_ROOTS=/opt/cloudlink/data/imports
CLOUDLINK_ALLOW_INSECURE_WORKER_INSTALL=0
WORKER_SECRET=$WORKER_SECRET
INTERNAL_API_SECRET=$INTERNAL_API_SECRET
CLOUDLINK_CODEX_TOKEN=$CLOUDLINK_CODEX_TOKEN
ADMIN_USERNAME=admin
ADMIN_PASSWORD=$ADMIN_PASSWORD
TASK_LOCK_SECONDS=1800
TASK_MAX_RETRIES=1
CLOUDLINK_MAX_PENDING_TASKS=10
CLOUDLINK_QUEUE_TIMEOUT_SECONDS=21600
CLOUDLINK_STARVATION_PROTECTION_SECONDS=900
CLOUDLINK_MINIMUM_WORKER_VERSION=2026.07.05.6
WORKER_ONLINE_SECONDS=180
TASK_ALLOWED_TYPES=echo_test,generate_daily_report,script_job
WORKER_INSTALL_INVITE_TTL_MINUTES=30
EOF
sudo chmod 600 /etc/cloudlink.env
printf 'Initial Cloudlink dashboard password: %s\n' "$ADMIN_PASSWORD"
```

Install the systemd service:

```bash
sudo tee /etc/systemd/system/cloudlink.service >/dev/null <<'EOF'
[Unit]
Description=Cloudlink task queue
After=network.target

[Service]
WorkingDirectory=/opt/cloudlink
EnvironmentFile=/etc/cloudlink.env
ExecStart=/opt/cloudlink/.venv/bin/python -m uvicorn app.main:app --host 127.0.0.1 --port 8010
Restart=always
RestartSec=3
User=ubuntu

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now cloudlink
```

For a domain such as `tasks.example.com`, point DNS to the server and proxy with
Nginx:

```nginx
server {
    listen 80;
    server_name tasks.example.com;

    location /api/internal/ {
        return 404;
    }

    location / {
        proxy_pass http://127.0.0.1:8010;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

After HTTPS is configured, set `CLOUDLINK_PUBLIC_BASE_URL` to the HTTPS origin
and restart `cloudlink.service`.

If there is no domain, set `CLOUDLINK_PUBLIC_BASE_URL` to a reachable IP-based
HTTPS URL and expose the reverse proxy carefully. Plain HTTP is acceptable only
for private testing when `CLOUDLINK_ALLOW_INSECURE_WORKER_INSTALL=1` is set; do
not use that mode on the public internet.

Create the limited Codex token file so cloud-side Codex CLI can use internal
APIs without reading `/etc/cloudlink.env`:

```bash
sudo awk -F= '/^CLOUDLINK_CODEX_TOKEN=/{print $2}' /etc/cloudlink.env > /opt/cloudlink/.codex-token
sudo chown ubuntu:ubuntu /opt/cloudlink/.codex-token
sudo chmod 600 /opt/cloudlink/.codex-token
```

Update from GitHub on the cloud server:

```bash
cd /opt/cloudlink
git fetch --tags origin
git checkout <branch-or-tag>
.venv/bin/python -m pip install -r requirements.txt
sudo systemctl restart cloudlink
sudo systemctl is-active cloudlink
```

After an update, the dashboard-generated worker installer automatically packages
the worker code from the currently deployed `/opt/cloudlink` tree, so new workers
match the deployed worker code. The server enforces
`CLOUDLINK_MINIMUM_WORKER_VERSION`, not exact equality with `CLOUDLINK_VERSION`.
Existing workers whose reported `cloudlink_version` is lower than the minimum are
shown as "需要更新" and cannot claim new tasks until they are updated from the
console deployment command. Server-only releases can bump `CLOUDLINK_VERSION`
without changing `CLOUDLINK_MINIMUM_WORKER_VERSION`, so compatible workers keep
running.

## Cloudflare Domain

Use a subdomain such as:

```text
tasks.example.com
```

In Cloudflare, point the subdomain to the cloud server and keep HTTPS enabled. On the server, put Nginx or another reverse proxy in front of Uvicorn:

```text
https://tasks.example.com -> http://127.0.0.1:8010
```

Only these routes need to be reachable through the public domain:

- `/` and `/api/admin/overview` for the password-protected dashboard.
- `/api/worker/*` for registered local workers.
- `/install/worker/*` for short-lived worker install scripts and packages.

For stricter isolation, block `/api/internal/*` in Nginx for public requests and call it only from server-side code through `http://127.0.0.1:8010`.

## Cloud-Side Codex CLI Access

Cloud-side Codex CLI runs in a restricted sandbox and should not use `sudo` to read `/etc/cloudlink.env`. Give it a limited local token instead:

```bash
sudo awk -F= '/^CLOUDLINK_CODEX_TOKEN=/{print $2}' /etc/cloudlink.env > /opt/cloudlink/.codex-token
sudo chown ubuntu:ubuntu /opt/cloudlink/.codex-token
sudo chmod 600 /opt/cloudlink/.codex-token
```

The token can create/query its own tasks, register/list datasets, and query `/api/internal/status` or `/api/internal/queue/status` through `127.0.0.1`. It receives aggregate resource occupancy, but not other submitters' task details. It can register datasets only from `CLOUDLINK_ALLOWED_DATASET_SOURCE_ROOTS`. It cannot delete datasets, access worker APIs, or access the admin dashboard.

## Queue And Scheduling Limits

Cloudlink protects local workers from unbounded task buildup:

- `CLOUDLINK_MAX_PENDING_TASKS` limits the number of pending tasks. The default is `10`.
- `CLOUDLINK_QUEUE_TIMEOUT_SECONDS` expires tasks that wait too long before being claimed. The default is `21600` seconds.
- `CLOUDLINK_STARVATION_PROTECTION_SECONDS` helps older large tasks avoid being starved by a constant stream of smaller tasks.
- Script jobs use `--timeout` for execution time after a worker starts the Python script. This is separate from queue wait time.
- Workers whose `cloudlink_version` is below `CLOUDLINK_MINIMUM_WORKER_VERSION` are marked as needing an update and cannot claim new tasks.

Cloud-side Codex CLI should inspect queue facts before batch submission:

```python
import sys

sys.path.insert(0, "/opt/cloudlink/scripts")
from cloudlink_client import request_json

queue = request_json("GET", "/api/internal/queue/status")
own_tasks = request_json("GET", "/api/internal/tasks")
```

Internal task lists are scoped to the current Codex token submitter, while queue
and resource status include aggregate occupancy. A Codex token can cancel its own
obsolete pending tasks:

```python
request_json(
    "POST",
    "/api/internal/tasks/<task_id>/cancel",
    {"reason": "obsolete pending task"},
)
```

## Add A Local Worker From The Dashboard

Workers must be registered before they can claim tasks. In normal operation,
do not manually copy worker files or manually register workers.

Open the dashboard, choose **本地算力节点 -> 添加节点**, select macOS, Linux, or
Windows, then copy the generated command to the local compute node terminal.

The generated command:

1. Downloads an OS-specific bootstrap script from `/install/worker/<token>/`.
2. Downloads `package.tar.gz` from the Cloudlink server, not from GitHub.
3. Verifies the worker package SHA256 before extracting it.
4. Installs the worker under the local user's Cloudlink directory.
5. Creates a Python virtualenv and installs requirements.
6. Exchanges the short-lived invite token for worker registration over HTTPS.
7. Writes local worker configuration with restricted file permissions and starts the worker.

The command contains only a short-lived invite token. It does not display
`WORKER_SECRET` in the dashboard. The worker receives a node-specific credential
only during the HTTPS registration exchange, and that credential is stored as a
hash on the server. `WORKER_SECRET` remains only as a compatibility fallback for
older manually registered workers that do not yet have a node-specific secret.

## Create An `echo_test` Task

Task creation is internal-only. Run this from the cloud server or trusted server-side code:

```bash
curl -X POST http://127.0.0.1:8010/api/internal/tasks \
  -H "Content-Type: application/json" \
  -H "X-Internal-API-Secret: $INTERNAL_API_SECRET" \
  -d '{"type":"echo_test","payload":{"message":"hello"}}'
```

The response includes an `id`.

Restricted Codex CLI scripts can use the local token helper without printing secrets:

```python
import sys

sys.path.insert(0, "/opt/cloudlink/scripts")
from cloudlink_client import request_json

status = request_json("GET", "/api/internal/status")
created = request_json(
    "POST",
    "/api/internal/tasks",
    {"type": "echo_test", "payload": {"message": "hello"}},
)
```

## Submit A Dynamic Script Job

This is the path intended for Codex CLI running on the cloud server. The cloud-side Codex CLI generates a concrete Python script, submits it to Cloudlink as one whitelisted task type, `script_job`, waits for the local compute node to finish, then continues analysis from the returned result.

The local worker does not call Codex CLI. It runs the submitted Python script directly in a Cloudlink-only virtualenv:

```text
~/.cloudlink/
  jobs/<task-id>/
    main.py
    inputs/
    outputs/
  venvs/python-auto/
```

If the payload declares Python requirements, the worker installs missing packages into `CLOUDLINK_PYTHON_AUTO_VENV` only. It does not install packages into system Python or the project `.venv`.

Files written under `outputs/` are returned in the task result. Small text files include `content`; binary files include `content_base64`; larger files are uploaded as retained result artifacts.

## Result Artifacts

Small files under `outputs/` are returned inline in `result.output_files`.
Large files are uploaded by the worker to `/opt/cloudlink/data/artifacts/tasks/<task_id>/`.
Artifact uploads use resumable chunks controlled by `CLOUDLINK_ARTIFACT_CHUNK_BYTES`
and finish with a server-side size/sha256 check. If a chunk request times out or
hits a transient server error, the worker queries upload status and continues
from the server-confirmed offset instead of treating the task as failed
immediately. Artifact uploads use a longer retry window than heartbeat and claim
requests: `CLOUDLINK_ARTIFACT_UPLOAD_RETRIES`,
`CLOUDLINK_ARTIFACT_RETRY_BASE_SECONDS`, and
`CLOUDLINK_ARTIFACT_RETRY_MAX_SECONDS`.

Use task `title`, `description`, `payload.task_context`, and `payload.expected_artifacts`
so humans and cloud-side Codex CLI can understand what each result file means.

Cloud-side Codex CLI can download uploaded artifacts through:

`GET /api/internal/tasks/<task_id>/artifacts/<artifact_id>/download`

## Manage Large Datasets

Cloudlink can maintain server-local data for reuse by local workers. Do not embed large K-line, tick, model, or archive data into a task payload.

Plain server files are managed as symlinks:

```bash
python3 - <<'PY'
import sys

sys.path.insert(0, "/opt/cloudlink/scripts")
from cloudlink_client import request_json

body = {
    "name": "example-prices",
    "version": "2024-v1",
    "title": "Example price CSV",
    "description": "Server-local CSV managed by symlink.",
    "source_kind": "symlink_file",
    "source_path": "/opt/cloudlink/data/imports/prices.csv",
    "content_type": "text/csv",
    "manifest": {"schema": ["timestamp", "open", "high", "low", "close", "volume"]},
}
print(request_json("POST", "/api/internal/datasets", body))
PY
```

Archives are moved into Cloudlink ownership:

```json
{
  "source_kind": "owned_archive",
  "source_path": "/tmp/klines.zip",
  "archive_format": "zip",
  "extract_required": true
}
```

Reference a managed dataset from a script job:

```bash
scripts/submit_local_script_job.py \
  --script-file /tmp/analyze.py \
  --dataset <dataset_version_id>:klines \
  --wait
```

The worker exposes the local path to the script:

```python
import os
from pathlib import Path

klines_path = Path(os.environ["CLOUDLINK_DATASET_KLINES"])
```

The dashboard shows server-managed datasets and which workers have cached or extracted them. Server deletion removes only Cloudlink symlinks for `symlink_file`; it removes real Cloudlink-owned archives for `owned_archive`. Worker-local deletion removes both downloaded archives and extracted directories.

On the cloud server:

```bash
cd /opt/cloudlink
source /etc/cloudlink.env

scripts/submit_local_script_job.py --wait --script '
from pathlib import Path

total = sum(i * i for i in range(5_000_000))
Path("outputs/result.txt").write_text(str(total), encoding="utf-8")
print(total)
'
```

For generated scripts saved as files:

```bash
scripts/submit_local_script_job.py \
  --script-file /tmp/cloudlink-heavy-job.py \
  --requirement pandas==2.2.2 \
  --requirement numpy==2.0.2 \
  --timeout 3600 \
  --wait
```

For scripts that need stdin, command-line args, or text input files:

```bash
scripts/submit_local_script_job.py \
  --script-file /tmp/analyze.py \
  --stdin-file /tmp/request.json \
  --arg=--fast \
  --input-file inputs/config.json=/tmp/config.json \
  --wait
```

Only code running on the server should call `/api/internal/tasks`; public Nginx should keep `/api/internal/*` blocked. Cloudlink also rejects internal API calls unless they arrive through a direct local request such as `127.0.0.1`.

## Worker API Auth

Worker endpoints require:

```http
Authorization: Bearer <worker-specific-secret>
```

Missing or incorrect credentials return `401`.

Worker IDs must also be registered and enabled in `worker_nodes`; otherwise worker APIs return `403`. For newly installed workers, the bearer token must match that worker's stored secret hash, so one worker cannot impersonate another worker ID. Older workers without a stored hash can still use the legacy `WORKER_SECRET` until they are reinstalled from the dashboard.

## Operations Dashboard

Open:

```text
https://tasks.example.com/
```

Use `ADMIN_USERNAME` and `ADMIN_PASSWORD` for the first Basic Auth login. The
dashboard can then change the admin password from **修改密码**. Changed passwords
are stored as PBKDF2 hashes in SQLite and take precedence over the environment
password.

The dashboard shows task counts, recent tasks, task details, result artifacts,
registered compute-node state, worker settings, managed datasets, local dataset
caches, and generated worker install commands.

If the dashboard password is lost, reset it on the cloud server by deleting the
stored hash; Cloudlink will fall back to `ADMIN_PASSWORD` from
`/etc/cloudlink.env`:

```bash
cd /opt/cloudlink
source /etc/cloudlink.env
sqlite3 "$CLOUDLINK_DATABASE_PATH" "DELETE FROM admin_credentials WHERE username = '$ADMIN_USERNAME';"
sudo systemctl restart cloudlink
```

## Troubleshooting Tasks Not Being Claimed

- Confirm the worker was installed or reinstalled from **本地算力节点 -> 添加节点** so it has a node-specific credential.
- For legacy/manual workers, confirm the backend and worker still share the fallback `WORKER_SECRET`.
- Confirm the worker is registered and enabled in `worker_nodes`.
- Confirm `WORKER_SUPPORTED_TYPES` includes the task type.
- Confirm `TASK_ALLOWED_TYPES` includes the task type on the server.
- Query the dashboard and check whether the worker is online.
- Run `scripts/start_local_worker.sh doctor` to isolate public HTTPS, worker heartbeat, dataset maintenance, or safe claim failures without claiming real tasks.
- If the worker process is still running but the dashboard shows offline, check for repeated network timeouts. The worker sends heartbeats in a background thread; `WORKER_API_TIMEOUT_SECONDS`, `WORKER_API_RETRIES`, `WORKER_API_RETRY_BASE_SECONDS`, `WORKER_API_RETRY_MAX_SECONDS`, `WORKER_HEARTBEAT_SECONDS`, and server `WORKER_ONLINE_SECONDS` control how quickly the dashboard reacts to public HTTPS interruptions.
- If no task is claimed, remember that the worker now tries task claim first and only checks dataset delete requests on `WORKER_MAINTENANCE_INTERVAL_SECONDS`, so a dataset-maintenance timeout should no longer block every idle poll.
- If a legacy start script fails while fetching `WORKER_SECRET` over SSH, rerun once when SSH is reachable. The start script caches the secret at `CLOUDLINK_WORKER_SECRET_FILE`, and later starts use the local cache instead of SSH. Dashboard-installed workers do not need SSH access to fetch the worker credential.
- Query the task and check whether `status` is `pending`, `running`, or `timeout`.
- If a worker crashed while running a task, wait for `TASK_LOCK_SECONDS`; the next worker claim can pick up the expired running task until `TASK_MAX_RETRIES` is reached.
- Check worker logs for network/API errors.

## Add A New Task Type

1. Add the type name to `TASK_ALLOWED_TYPES`.
2. Add it to `WORKER_SUPPORTED_TYPES` for workers that can run it.
3. Register the worker with the new type in `/api/internal/workers`.
4. Add a new explicit branch in `CloudWorker.run_task`.
5. Implement a Python handler function.
6. Add tests for task creation and worker behavior.

Do not add task types that execute arbitrary shell commands from payload data.
